Properties

Een klasse heeft over het algemeen private en public fields. Private fields kunnen alleen benaderd worden vanuit de klasse zelf. Public fields zijn ook toegankelijk van buitenaf.

Als je in java een field alleen leesbaar wil maken dan is het gebruikelijk om een private field te maken met een bijbehorende functie als 'getter'. Daarnaast is het mogelijk een functie als 'setter' toe te voegen en eventuele validatie uit te voeren op de nieuwe waarde van het veld.

In C# doen we dit net even anders. Hier hebben we namelijk een speciale syntax voor het maken van getters en setters. Dit zijn properties. Je bent bijna verplicht om in C# altijd properties te gebruiken in plaats van normale 'fields'. Sommige frameworks (waar we in de toekomst gebruik van gaan maken) werken bijvoorbeeld alleen goed samen met properties en niet met fields.

Java

private String myField; 

public String getMyField()
{
    return this.myField;
}
public void setMyField(String value)
{
     this.myField = value;
}

C#

public class Person
{
    private string _firstname;

    public string Firstname
    {
        get
        {
            return _firstname;
        }
        set
        {
            _firstname = value;
        }
    }
}

Dit kan ook in een kortere variant. In het onderstaande voorbeeld zal er onder water een private field gemaakt worden waarmee de getter en setter hun waarde benaderen. Het is slim om jezelf aan te leren deze korte syntax te gebruiken voor properties i.p.v public fields. Hiermee kun je in de toekomst makkelijker bijvoorbeeld de 'set' private maken.

Daarnaast is het ook mogelijk om zelf te bepalen of je een get of een set implementeert. De implementatie mag ook meer logica bevatten.

Korte syntax

public class Person
{
    private string _firstname;

    //De lange variant (ingeklapt)
    public string Firstname...
	
    //De korte variant
    public string Lastname { get; set; }
	
}

Logica

public class Person
{
    private string _firstname;

    //De lange variant (ingeklapt)
    public string Firstname..

    public string Lastname { get; set; }

    public string Fullname 
    { 
        get
        {
            return _firstname + Lastname;
        }
    }
}
Lambda Expressies
"A lambda expression is an anonymous function that you can use to create delegates or expression tree types. By using lambda expressions, you can write local functions that can be passed as arguments or returned as the value of function calls. Lambda expressions are particularly helpful for writing LINQ query expressions.""

Duidelijk toch? Dat dacht ik al

lambda expressies worden gebruikt om Linq queries (zie volgende blokje) te schrijven en om deze toe te kunnen passen is de basiskennis nodig. Laten we beginnen met de vorm van een lambda expressie. 

	n => n % 2 == 1

Wat doet deze expressie nu eigenlijk? Als je een lambda expressie definiëert, definiëer je onder water een 'anonieme' functie. Dit is een functie zonder naam, die maar 1 keer uitgevoerd wordt. Lambda expressies zijn niet meer dan een 'korte syntax' om snel even een functie te definiëren en uit te voeren. De bovenstaande lambda expressie bestaat uit de volgende onderdelen:

  • n is de input parameter (parameters van de functie)
  • n % 2 == 1 is de expressie (inhoud van de functie)

Deze expressie vertelt ons niet zo veel. Maar laten we nu een wat realistischer scenario nemen. Stel, je wilt een lijst filteren. Normaal zou je dan een for loop definiëren, door alle elementen van je collectie loopen en je filter (eventueel functie) toepassen.

Een lambda expressie kan dit voor ons een stuk simpeler maken.

Old scewl

public void DoSomething()
{
    List lijstje = new List();
    List gefilterd = new List();

    gefilterd = lijstje.Where(LastnameSmulders).ToList();
}

public bool LastnameSmulders(Person person)
{
    return person.Lastname == "Smulders";
}

Lambda

List lijstje = new List();
List gefilterd = new List();

gefilterd = lijstje.Where(p => p.Lastname == "Smulders").ToList();

De 'Where' functie op de lijst is onderdeel van Linq. Hier gaan we het zo over hebben. Deze vervangt voor ons de 'for loop'.

Onze lambda statement vervangt in dit voorbeeld de functie "LastnameSmulders". de p die we meegeven is de naam van de parameter die onze 'anonieme functie' meekrijgt. De 'Where' statement roept onze anonieme functie aan met elke keer 1 persoon uit de lijst. Onze expressie wordt omgezet in een boolean. Je zou dit dus als een 'if' statement kunnen lezen. Alle personen waarbij deze expressie 'true' is worden verzameld in een nieuwe lijst als return value van de functie Where.

Als je vaker gebruik gaat maken van lambda expressies, voelt de syntax vanzelf vertrouwd aan. lambda expressies kunnen je een hoop tijd besparen!

Het meegeven van een functie als 'parameter' valt onder de categorie 'delegates'. Hier gaan we later dit blok nog op in.

Linq

Genoeg gebazel over lambda expressies. Laten we gewoon code gaan tikken!

Voor we de nieuwe stijl demonstreren, eerst even een demonstratie hoe we het vroeger deden.

List result = new List();
foreach (Person person in People)
{
  if(person.FirstName[0] == firstChar)
    result.Add(person);
}
return result;

Nu gaan we gebruik maken van Linq. Linq is een assembly (system.linq) waarmee je filter-functies kunt uitvoeren op lijsten. Deze functies kun je uitvoeren doormiddel van een linq query of een lambda expressies. 

using System.Linq;

Deze Linq query's maken je leven een stuk makkelijker! Deze voorbeelden zijn maar het topje van de ijsberg wat je allemaal kan met Linq , en ik kan je aanraden om ook even op het wereld wijde web verder te zoeken!

Linq query

	
return from p in People where p.FirstName[0] == firstChar select p;

Linq met Lambda

	
return People.Where(p => p.FirstName[0] == firstChar);
Let op!

De return value van een linq statement is van het type IEnumerable. Dit is een ander type dan List. Wil je toch een List als result? 

lijstje.Select(p => p.Lastname).toList();

Coding Guidelines

Coding Guidelines houdt in dat we afspraken gaan maken over hoe onze code 'stylen' Hierbij moet je denken aan bijvoorbeeld de volgende dingen:

  • Inspringen
  • Haakjes
  • Wanneer een lege regel
  • Wanneer & hoe commentaar

Het maakt niet uit welke coding guidelines je kiest als iedereen binnen je team maar dezelfde guidelines hanteert. Bijvoorbeeld de microsoft coding guidelines.

Microsoft Coding guidelines
https://msdn.microsoft.com/en-us/library/ff926074.aspx

Daarnaast is het belangrijk om je te houden aan de coding conventions. Coding conventions zijn afspraken over de naamgeving binnen C#. In ons geval zijn dit PascalCasing en camelCasing.

PascalCasing ( NumberOfWheels )

PascalCasing houdt in dat we de naam van een variabele beginnen met een hoofdletter. Verder beginnen alle lossen woorden binnen de naam ook met een hoofdletter. PascalCasing passen we toe op de volgende onderdelen:

  • Classes
  • Namespaces
  • Methods
  • Constructors / Deconstructors
  • Properties

camelCasing ( numberOfWheels )

camelCasing is ongeveerd het zelfde als PascalCasing, op het feit na dat de eerste letter van de naam een kleine letter is. camelCasing passen we toe op de volgende onderdelen:

  • Fields
  • Lokale variabelen
  • Method parameters

Unit Testing
" Unit Testen... Hmmm, is dat niet het onderdeel van software ontwikkeling dat ik altijd oversla? "

Komt deze tekst je bekend voor? Dan hoop ik dat ik je van mening kan laten veranderen.

Heb je wel eens een nieuw stuk code geschreven, waarmee je een klein beetje nieuwe logica introduceerde?
Zat dit stukje logica in een scherm waar je naartoe moest navigeren?
Blijkt dan keer op keer dat je nog net een klein dom foutje nog in de code hebt zitten?

Elke keer handmatig je code testen kost zeeën van tijd. Je kunt ook een unit test schrijven, om op deze manier de code aan te roepen en te testen. Als je dan denkt dat je klaar bent, hoef je het nog maar 1x in de applicatie te testen. Daarnaast kun je deze code later in het ontwikkelproces (gratis) automatisch opnieuw testen.

Daarnaast zijn er foutjes in je code waarbij je er pas heel laat achter komt dat er iets fout gaat. Een snelle berekening om een aantal waardes op te tellen bijvoorbeeld. Op je scherm zie je een getal staan, maar dit betekent dit dat je code juist is? Een Unit test schrijven om te verifiëren dat de uitkomst van je berekening ook echt klopt, kan je letterlijk uren besparen.

Maar als je mij niet gelooft, zijn er genoeg andere redenen om unit-test te schrijven!

In Visual Studio kun je een Unit Test Project toevoegen aan je solution, waar je kleine stukjes code kan schrijven die methodes voor je kunnen aanroepen en de output van deze methodes kunnen 'asserten'.

Dit project bestaat uit classes met een TestClass attribuut, alswel methodes met een TestMethod attribuut. Deze methodes kun je uitvoeren via de Test Window in visual studio. Hier kun je zien of een test geslaagd is en zo niet, wat dan de reden is dat de test niet gelukt is.

Tip: De test explorer kun je vinden onder Test > Window > Test Explorer

Voordat je kan beginnen met het schrijven van de test, moet je eerst een duidelijke naam verzinnen voor jouw unit test. Als een test mislukt, wil je meteen kunnen zien waar in de code er iets fout is gegaan. Gebruik de volgende regels als houvast:

  1. De naam van de TestClass = de naam van de klasse die je gaat testen + "Test"
      bijv. CarTest
  2. De naam van de TestMethode moet uit de volgende onderdelen bestaan:
    • Naam van de klasse
    • Naam van de methode die je gaat testen (of bijv. constructor)
    • Of de test moet slagen, of als je verwacht dat er exception komt
        • bijv. CarTest_GetGasoline_Succes

Unit Testing - Triple AAA

Een unit test schrijven bestaat uit 3 stappen:

  1. Arrange
  2. Act
  3. Assert

De eerste twee stappen spreken voor zich. Bij de Arrange creëer je de klassen die je nodig hebt voor het uitvoeren van je test. Daarnaast kun je een aantal andere objecten klaar zetten die je nodig hebt. Bij de Act ga je letterlijk de methode die je wilt testen, uitvoeren. Om het resultaat van de methode te verifiëren gaan we gebruik maken van de Assert klasse in .NET.

Het .NET framework biedt je toegang tot de Assert klasse. Met deze klasse kun je waardes vergelijken. Komen de waardes niet overeen, dan mislukt je unit test.

Assert

In your test method, you can call any number of methods of the Assert class, such as Assert.AreEqual(). The Assert class has many methods to choose from, and many of those methods have several overloads. 
//1. arrange
Calculator calcy = new Calculator();

//2. act
var result = calcy.Add(1, 1);

//3. assert
Assert.AreEqual(2, result);
